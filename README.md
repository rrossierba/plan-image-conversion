# PDDL Plan-to-Image Converter

This project is a specialized utility designed to convert PDDL plans—specifically, lists of actions—into visual media formats (PNG, MP4, or GIF).

It is derived from the backend of the [Planimation](https://github.com/planimation/backend) project (documentation available [here](https://planimation.github.io/documentation/backend_dev_guide/)). 
While the original Planimation architecture functions as a web application that solves and visualizes single problem instances end-to-end, this project is adapted for **batch processing pre-computed plans**.

### Core Functionality & Scope

Unlike the standard Planimation workflow, which requires solving a single PDDL problem instance to generate a plan, this tool operates on plans that have already been generated by a solver. 
Its primary scope is the conversion of these existing plan objects (in conjunction with their relative PDDL domain and problem files) into visual representations.

The source code extracts and modifies core elements of the Planimation backend to facilitate:

1. **Batch Processing:** Converting multiple plans iteratively rather than handling a single user request.
2. **Direct Input:** Bypassing the solver stage to work directly with serialized plan objects.

The system utilizes an [Animation Profile](https://planimation.github.io/documentation/ap_guide/)—a custom PDDL-based file that specifies visual rules for domain predicates and visual assets for objects—to generate a Visualization File Generator (VFG). 
The VFG maps the state of objects at each plan stage, which is then rendered into the final media output.

## Configuration

To use the converter, a configuration file (JSON) is required. 
A template is provided at `config_template.json`. The configuration must specify the following parameters:

* **`output_format`**: The desired output format (`png`, `mp4`, or `gif`).
* **`domain_name`**: The PDDL domain being visualized. Currently supported domains are `BLOCKSWORLD` and `LOGISTICS` (in progress).
* **`results_path`**: The destination directory for the generated media.
* **`domain_file`**: Path to the PDDL domain file.
* **`animation_profile`**: Path to the PDDL animation profile file.
* **`problems_path`**: Directory containing the PDDL problem instances associated with the plans.
* **`plans_path`**: Path to the `.pkl` plan object (or a list of objects) to be converted.
* **`num_cores`**: The number of CPU cores to utilize for parallel processing.

## Usage

The application can be executed either via a Docker container (recommended) or as a standalone Python script. 
The source code is located in the `src/` directory, with `main.py` serving as the entry point.

### Method 1: Docker (Recommended)

Using Docker ensures all dependencies are correctly managed.

**1. Environment Setup (Linux/macOS)**
To prevent permission issues when the container writes files to your host system, it is recommended to create a `.env` file. This file should define the user ID and group ID of the host user running the script.

Create a file named `.env` in the root directory:

```bash
USER_ID=1000        # Replace with your actual UID (run `id -u`)
GROUP_ID=1000       # Replace with your actual GID (run `id -g`)
USERNAME=user_name  # Replace with your username
```

**2. Build the Image**
Build the Docker image using the provided Dockerfile:

```bash
docker build -t plan-image-conversion .
```

**3. Run the Container**
When running the container, mount the necessary directories and pass the environment file. Ensure your configuration file points to the mapped paths inside the container (e.g., `/app/files`).

```bash
docker run \
  --env-file .env \
  -v "$(pwd)/files":/app/files \ 
  -e CONFIG_FILE=/app/files/config.json \
  -d \
  --name converter \
  plan-image-conversion
```

### Method 2: Local Python Execution

To run the script directly on your machine:

1. **Environment Setup:** Create a virtual environment using [venv](https://docs.python.org/3/library/venv.html) or [uv](https://docs.astral.sh/uv/).
2. **Dependencies:** Install the required packages found in `src/requirements.txt`.
3. **Execution:** Run the `main.py` script, passing the path to your configuration file via the `--config` argument.

```bash
python src/main.py --config path/to/your/config.json
```

## How it works

The architecture is designed to decouple the conversion orchestration from the domain-specific visualization logic. 
The process is divided into four main stages:

### 1. Initialization and Configuration (`main.py`)

The execution begins by parsing the configuration file (`config.json`). 
Based on the specified `domain` (e.g. `blocksworld` or `logistics`), the script dynamically selects the appropriate implementation classes using a factory pattern:

* **`Visualizer` Class**: Handles the rendering logic and media export.
* **`VfgPlan` Class**: Manages the interaction with the Planimation backend parsers.
* **Name Parser**: A specific function to map the plan name to its corresponding PDDL problem file.

These components are injected into the `GenericConverter`, ensuring the core conversion logic remains domain-agnostic.

By now the only implemented domains are `blocksworld` and `logistics` (which is still in progress).

### 2. Orchestration and Parallelization (`convert.py`)

The `GenericConverter` manages the batch processing of plans. It leverages Python's `multiprocessing` module to maximize throughput (number of jobs is a configuration parameter).

* **Global State Management**: To optimize memory usage during parallel execution, heavy read-only assets (domain text, animation profile patterns, and configuration paths) are initialized once globally for all worker processes via the `worker_initializer`.
* **Plan Distribution**: The list of plan objects (loaded from `.pkl` files) is distributed across a pool of worker processes.

### 3. Logic Adaptation (`vfg_plan.py`)

For each plan, a `VfgPlan` object is instantiated. 
This class acts as an adapter between the raw PDDL data, the plan (Action list) and the Planimation media creation engine.

* It utilizes internal parsers (`get_domain_json`, `get_problem_dic`, `get_stages`) to process the PDDL domain, problem instance, and the action list.
* It compiles this data into a **VFG (Visualization File Generator)** structure, which describes the state of every object at every step of the plan.
* Domain-specific subclasses (e.g., `BlocksWorldVfgPlan`) implement auxiliary methods, such as calculating scene dimensions based on the number of objects involved.

### 4. Rendering and Export (`visualizer.py`)

The `Visualizer` class receives the processed VFG data and handles the final output generation calling.

* **Dynamic Profiling**: Before rendering, the `adjust_animation_profile` method dynamically modifies the PDDL Animation Profile. For domains like Blocksworld, it calculates visual parameters (block size, claw position, spacing) relative to the image resolution (`dpi`, `figsize`) and problem complexity.
* **Media Generation**: The system generates the visualization frames using the `export_media` utility.
* **Format Handling**:
* **Images (PNG)**: results are extracted from the generated zip archive and saved as a sequence of images in a specific folder.
* **Video (MP4/GIF)**: the binary stream is written directly to the output file.
